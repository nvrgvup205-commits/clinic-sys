# مرجع نظام حُلول Hulool - دليل شامل للذكاء الصناعي والمطورين

> آخر تحديث: 29 مايو 2026
> الغرض: ملف مرجعي كامل يمكن استخدامه مع أي AI لفهم المشروع والاستمرار في التطوير

---

## 1. نظرة عامة على المشروع

### ما هو حُلول؟
نظام SaaS لإدارة العيادات الطبية في السعودية، يدعم كل التخصصات (أسنان، أطفال، جلدية، عيون، باطنة، نساء وولادة، إلخ).

### التقنيات المستخدمة
```
Frontend: React + Vite + Tailwind CSS
Backend: Supabase (Database + Auth + Storage + Realtime)
Deployment: Railway / Vercel
```

### الروابط
```
/                         صفحة الهبوط الرئيسية
/owner                    لوحة تحكم المالك (صاحب النظام)
/{clinic-slug}            بوابة المريض
/{clinic-slug}/about      معلومات العيادة
/{clinic-slug}/staff      بوابة الموظفين (أدمن/استقبال/محاسب/دكتور)
```

## 2. الأدوار والصلاحيات

### المالك (Owner)
- صاحب النظام كله
- يدخل من `/owner`
- يرى كل العيادات المشتركة في النظام

### الأدمن (Clinic Admin)
- صاحب/مدير العيادة
- يدخل من `/{clinic-slug}/staff` → اختيار "أدمن"
- صلاحياته:
  - إدارة الأطباء والموظفين
  - إدارة المرضى
  - إدارة الخدمات والأسعار (هو الوحيد اللي يغيّر الأسعار)
  - رؤية التقارير والإحصائيات
  - إعدادات العيادة

### الاستقبال (Receptionist)
- يدخل من `/{clinic-slug}/staff` → اختيار "استقبال"
- **لازم يبدأ الدوام أولاً قبل ما يشوف أي شي**
- صلاحياته:
  - تأكيد الحجوزات
  - البحث عن المرضى (بالجوال/الهوية/الاسم)
  - تسجيل دفع الكشف الأساسي
  - استقبال طلبات الدفع من الدكتور وتسجيلها
  - **لا يغيّر الأسعار أبداً**
  - رؤية الأطباء النشطين

### الطبيب (Doctor)
- يدخل من `/{clinic-slug}/staff` → اختيار "دكتور"
- **لازم يبدأ الدوام أولاً**
- **لا يرى المريض إلا بعد ما يدفع عند الاستقبال**
- صلاحياته:
  - بدء الكشف وتسجيل التشخيص والعلاج والوصفة
  - رفع مرفقات الكشف (أشعة/تحاليل/تقارير)
  - طلب خدمة إضافية من الاستقبال
  - **لا يحصّل أي فلوس**
  - **لا ينهي الكشف لو فيه طلب دفع معلّق**

### المحاسب (Accountant)
- يدخل من `/{clinic-slug}/staff` → اختيار "محاسب"
- صلاحياته:
  - رؤية الإيرادات والمصروفات والأرباح
  - رؤية الفواتير والمدفوعات
  - تقارير ساعات عمل الموظفين
  - استيراد ملفات Excel
  - تصدير التقارير

## 3. سير العمل (Workflow)

### الدفع الأساسي
```
1. المريض يحجز موعد
2. الحجز يظهر للاستقبال (Realtime)
3. الاستقبال يؤكد الحجز
4. المريض يجي العيادة
5. الاستقبال يسجّل دفع الكشف الأساسي
6. المريض يظهر للدكتور (Realtime)
7. الدكتور يبدأ الكشف
```

### طلب خدمة إضافية
```
1. الدكتور يختار خدمة إضافية من القائمة
2. النظام ينشئ "طلب دفع" (payment_request) بحالة pending
3. الطلب يظهر للاستقبال (Realtime)
4. الاستقبال يسجّل الدفع (بدون تغيير السعر)
5. الدكتور يشوف "تم الدفع" (Realtime)
6. الدكتور يكمل الكشف
```

### قاعدة ذهبية
```
الاستقبال لا يغيّر الأسعار
الدكتور لا يغيّر الأسعار ولا يحصّل فلوس
المحاسب لا يغيّر الأسعار
الأدمن فقط يغيّر الأسعار
```

## 4. نظام الدوام (Work Sessions)

### القواعد
- الاستقبال والدكتور لازم يضغطوا "بدء الدوام" قبل ما يشوفوا أي تفاصيل
- عند الخروج يتوقف حساب الوقت
- لو خرج بالغلط، الوقت بعد الخروج لا يحسب
- نفس الشخص لا يفتح جلستين دوام في نفس الوقت
- أكثر من شخص في نفس الدور يقدرون يشتغلون معاً

### حالة نشط/غير نشط
```
نشط = عنده جلسة مفتوحة (clock_out = null)
غير نشط = ما عنده جلسة مفتوحة
```

## 5. Realtime (التحديث اللحظي)

### كل شي لازم يتحدث لحظياً:
- حجز مريض → يظهر للاستقبال فوراً
- تأكيد الحجز → يظهر للمريض فوراً
- دفع الكشف → يظهر للدكتور فوراً
- طلب دفع من الدكتور → يظهر للاستقبال فوراً
- تسجيل الدفع → يظهر للدكتور فوراً
- دخول/خروج موظف → يظهر للأدمن والمحاسب فوراً

### الجداول اللي تحتاج Realtime
```
appointments
payment_requests
payments
invoices
medical_records
complaints
work_sessions
patients
```

### فلترة Realtime (مهم جداً)
```javascript
// لازم تفلتر بـ clinic_id عشان ما تحمّل بيانات كل العيادات
.eq('clinic_id', clinic.id)
```

## 6. قاعدة البيانات (Database Schema)

### الجداول الأساسية
```sql
clinics              -- العيادات
admin_users          -- الموظفين (أدمن/استقبال/محاسب)
doctors              -- الأطباء
patients             -- المرضى
appointments         -- المواعيد
clinic_services      -- الخدمات والأسعار
medical_records      -- السجلات الطبية
complaints           -- الشكاوى
expenses             -- المصروفات
```

### الجداول الجديدة (من الترقية)
```sql
payment_requests     -- طلبات الدفع من الدكتور للاستقبال
work_sessions        -- جلسات الدوام
invoices             -- الفواتير
payments             -- المدفوعات
```

### أعمدة مهمة في appointments
```sql
doctor_started_at    -- وقت بدء الكشف
doctor_completed_at  -- وقت انتهاء الكشف
paid_at              -- وقت دفع الكشف الأساسي
paid_by              -- من سجّل الدفع
```

### عزل بيانات العيادات
```
كل جدول فيه عمود clinic_id
كل Query لازم يحتوي .eq('clinic_id', clinic.id)
كل Insert لازم يضيف clinic_id
```

## 12. تعليمات للذكاء الصناعي

### قبل أي تعديل
1. لا تنفذ أي شي بدون إذن صريح من المستخدم
2. اسأل "نفّذ؟" وانتظر الموافقة
3. افهم السياق قبل التعديل

### قواعد مهمة
1. الأدمن = مدير/صاحب العيادة (مش مالك النظام)
2. المالك = صاحب النظام كله (صفحة مستقلة)
3. كل شي لازم Realtime
4. لا تسمح بتعديل الأسعار إلا للأدمن
5. الاستقبال لا يغيّر المبالغ
6. الدكتور لا يحصّل فلوس
7. افصل بيانات العيادات دائماً بـ clinic_id
8. استخدم اللهجة السعودية

### عند إنشاء ملفات جديدة
- أنشئ في مجلدات واضحة تبدأ بـ `00_`
- لا تحذف الملفات القديمة
- وثّق التغييرات

## 13. المشاكل الشائعة والحلول

### الموظف ما يقدر يدخل
- تأكد إن `is_active = true` في جدول admin_users أو doctors
- تأكد إن الدور (role) صحيح

### Realtime ما يشتغل
- تأكد إن الجدول مضاف في `supabase_realtime`
- تأكد إن الفلترة بـ clinic_id صحيحة

### الدكتور ما يشوف المريض
- المريض لازم يكون دفع الكشف (paid_at != null)

### طلب الدفع ما يظهر للاستقبال
- تأكد إن payment_requests مفعّل في Realtime
- تأكد إن الفلترة صحيحة

## 14. خطوات الترقية

### لترقية قاعدة البيانات
1. افتح Supabase → SQL Editor
2. انسخ محتوى ملف `HULOOL_UPGRADE_MIGRATION.sql`
3. الصقه واضغط Run
4. تجاهل أخطاء "already exists"

### للتحقق من النجاح
```sql
-- تأكد من وجود الجداول الجديدة
SELECT table_name FROM information_schema.tables 
WHERE table_schema = 'public' 
AND table_name IN ('payment_requests', 'work_sessions', 'invoices', 'payments');
```
